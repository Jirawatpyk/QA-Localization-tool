- `users` — has tenant_id; UPDATE by `users.id` alone is safe (Supabase Auth UID globally unique)
- `user_roles` — always filter by both userId + tenantId
- `projects`, `files`, `scores`, `findings`, `review_actions`
- `notifications`, `audit_logs`, `glossaries`
- `glossary_terms` — NO tenant_id column! Access via `glossary_id` FK → glossaries (which has tenant_id)
- `language_pair_configs`, `segments`, `review_sessions`
- `ai_usage_logs`, `ai_metrics_timeseries`, `audit_results`, `exported_reports`
- `feedback_events`, `file_assignments`, `fix_suggestions`, `run_metadata`
- `finding_assignments`, `finding_comments` — both have tenant_id NOT NULL + FK tenants (Story 5.2b confirmed)
- `self_healing_config`, `severity_configs` (nullable tenant_id = system default rows)
- `suppression_rules`

## Key Patterns to Watch

- `glossary_terms` has NO tenant_id — always access via verified glossaryId from glossaries table
- `severity_configs` has nullable tenant_id (system defaults have NULL) — query must handle this
- `taxonomy_definitions` is global — never add tenant filter (it would be wrong)
- `ai_usage_logs` is tenant-scoped — SELECT uses withTenant(); INSERT sets tenantId in values
- INSERT isolation pattern: no WHERE on INSERT — set `tenantId` in value object explicitly
- Inngest tenantId chain: `requireRole()` → Inngest event payload → typed parameter → every DB query
- ANTI-PATTERN: `withTenant()` not in `.where()` = dead code. Flag HIGH.
- ANTI-PATTERN: INSERT with unverified FK from client — always SELECT with withTenant() first.
- ANTI-PATTERN: over-broad DELETE in idempotent re-run — scope DELETE to specific layer/entity.
- LEFT JOIN rule: `withTenant()` on driving table goes in WHERE; on joined table goes in JOIN condition.
- Realtime rule: EVERY `postgres_changes` channel on a tenant-scoped table MUST include `filter: tenant_id=eq.${tenantId}`.
- Polling fallback rule: EVERY PostgREST polling query MUST include `.eq('tenant_id', tenantId)`.
- Realtime publication gap: tables must be added via `ALTER PUBLICATION supabase_realtime ADD TABLE t;` or channels silently never fire. Only `user_roles` confirmed published so far.
- Prop tenantId trust: hooks that accept tenantId as prop should derive from session or assert match.
- TOCTOU transaction pattern: SELECT then UPDATE inside same tx is safe, but the UPDATE must STILL include withTenant() as defense-in-depth. "Already verified via SELECT in same tx" is not sufficient justification to omit withTenant() on the UPDATE.
- Pure parsers (sdlxliffParser.ts, excelParser.ts): these are zero-DB-access utility functions. No tenant isolation checks needed. Confirm no DB import before marking PASS.
- Client stores (review.store.ts) and client hooks (use-undo-redo.ts, useFileUpload.ts): no DB access = no isolation risk. Focus audit on the server actions they delegate to.
- segments DELETE in batchInsertSegments uses `eq(segments.fileId, fileId)` without withTenant() — safe because fileId was already verified with withTenant() in the ownership SELECT upstream, and the whole thing runs inside a transaction. Acceptable pattern (same logic as TOCTOU note above, but here the DELETE is scoped to an already-verified FK).
- `back_translation_cache` table is tenant-scoped (has tenantId FK). Global cron TTL DELETE on it is intentional and acceptable — document with comment.
- Raw SQL subquery anti-pattern: when using `sql\`col IN (SELECT ... FROM t1 JOIN t2 ...)\``inside a WHERE, EVERY table in the subquery must also carry`AND t.tenant_id = ${tenantId}`. The outer `withTenant` on the primary table does NOT extend into subquery scope.
- Global cron Inngest functions (no event payload, no tenantId): cross-tenant DB operations are intentional — must be documented with a comment explaining cross-tenant scope to prevent future "missing withTenant" false fixes.
- RLS EXISTS subquery JOIN rule: every table aliased inside an EXISTS subquery must carry its own `AND alias.tenant_id = JWT_tenant_id` — even when the outer USING already binds the primary table. Outer USING does NOT extend into JOIN aliases inside the subquery. This mirrors the Drizzle "JOIN queries filter tenant_id on ALL joined tables" rule.
- `finding_assignments` Realtime publication: added to `supabase_realtime` in migration 00026. `finding_comments` intentionally NOT published (no live streaming in 5.2b scope).
