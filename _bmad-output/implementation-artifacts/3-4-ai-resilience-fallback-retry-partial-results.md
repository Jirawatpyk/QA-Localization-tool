# Story 3.4: AI Resilience — Fallback, Retry & Partial Results

Status: review

## Story

As a QA Reviewer,
I want AI processing to gracefully handle failures with fallback providers, automatic retries, and preserved partial results,
So that I never lose rule-based findings and can continue working even when AI services are degraded.

## Acceptance Criteria

1. **AC1 — Fallback chain consumption (L2):** When the primary L2 model (`gpt-4o-mini`) fails during a chunk's `generateText` call, the system automatically tries fallback models in order: (1) pinned version from project config, (2) system default, (3) `gemini-2.0-flash`. Currently `runL2ForFile.ts` calls `getModelForLayerWithFallback()` but only uses `.primary` (TD-PIPE-004). Each fallback attempt is logged: `{ fallback: true, originalModel, actualModel, reason, layer: 'L2' }`. If all models fail for a chunk, that chunk is marked as failed (existing partial failure handling continues).

2. **AC2 — Fallback chain consumption (L3):** Same pattern as AC1 but for L3: primary `claude-sonnet-4-5-20250929` → fallback `gpt-4o`. Each fallback attempt logged with audit flag. Findings from fallback models store the actual model used in `findings.ai_model` (already implemented — just verify). **Note:** Epic AC mentions "confidence recalibration per language pair" for fallback models — this is deferred to **Story 3.5 — Score Lifecycle & Confidence Display** (per-language calibration is confidence display scope, not fallback mechanics).

3. **AC3 — Fallback badge on findings:** Findings generated by a non-primary model show a visual indicator in `FindingListItem`. Detection: compare `finding.aiModel` against `LAYER_DEFAULTS[layer].systemDefault` (or project's pinned model). If different → render amber badge: "Fallback" with tooltip: "Generated by {model} fallback — confidence may differ from primary model". The comparison data is available at render time from the finding's `aiModel` field + a static lookup of default models.

4. **AC4 — Partial results when AI fails:** When ALL AI models are exhausted AND Inngest retries are exhausted for L2:
   - File status set to `ai_partial` (new status value in `DbFileStatus`)
   - All L1 findings remain intact and visible
   - Score calculated using L1 findings only
   - `scores.score_status` = `'partial'` (already exists in ScoreStatus type)
   - `scores.layer_completed` = `'L1'` (already exists)
   - ScoreBadge shows new `'partial'` state: yellow color, label "Partial — AI unavailable"

5. **AC5 — L2 succeeds but L3 fails (Thorough mode):** When L3 processing fails after fallback chain exhaustion:
   - L1 + L2 findings preserved, score reflects L1+L2
   - `scores.layer_completed` = `'L1L2'` with `scores.score_status` = `'partial'`
   - File status = `ai_partial` (not `failed`)
   - ScoreBadge shows `'partial'` state with yellow warning: "Deep analysis unavailable — showing screening results"

6. **AC6 — Retry AI Analysis:** A "Retry AI Analysis" button appears in `ReviewPageClient` when file has `ai_partial` status or `scores.score_status === 'partial'`:
   - Clicking triggers `retryAiAnalysis` server action
   - Server action queries `projects.processingMode` (current project setting) to determine expected layers. **Design decision:** Mode comes from current project config, NOT stored per-file/batch. If mode changed from Thorough→Economy after L3 failure, retry respects the new Economy mode (no L3 attempted). This is correct: current mode = current intent.
   - Layer detection: compare `scores.layer_completed` vs expected layers for current mode. If mode=economy and `layer_completed='L1'` → retry L2. If mode=thorough and `layer_completed='L1'` → retry L2+L3. If mode=thorough and `layer_completed='L1L2'` → retry L3 only. If mode=economy and `layer_completed='L1L2'` and `score_status='partial'` → no layers to retry, just recalculate score as `'calculated'`.
   - A new Inngest function `retryFailedLayers` re-runs only the failed layers
   - **CAS guard reset:** Before calling `runL2ForFile`/`runL3ForFile`, the retry function MUST reset file status to the expected pre-layer status (`l1_completed` for L2 retry, `l2_completed` for L3 retry) — this is a separate Inngest step for idempotency. The helper's CAS guard then checks the reset status normally.
   - Existing findings from successful layers are preserved (no DELETE of L1/L2 findings)
   - During retry: button disabled with spinner, ScoreBadge shows 'analyzing'
   - After success: normal score badge + findings update via Realtime
   - **Estimated wait time:** Deferred to Epic 4 — requires Inngest queue depth API integration. Current MVP: button shows "Retrying..." without time estimate.

7. **AC7 — Inngest retry configuration:** `processFilePipeline` updated to handle partial results at the handler level:
   - L2 step wrapped in try-catch **at handler level** (not inside step.run — Guardrail #13)
   - If L2 step throws after Inngest retries: set `ai_partial` status + calculate L1-only score
   - If L3 step throws (Thorough mode): set `ai_partial` + keep L1L2 score
   - `onFailureFn` updated: check which layers completed before setting status. If L1 done → `ai_partial` (not `failed`). If even L1 failed → `failed`

8. **AC8 — Batch completion with mixed status:** When processing a batch where some files succeed and others fail:
   - Each file's status tracked independently (already the case)
   - `ai_partial` counted as a terminal status for batch completion (alongside `l2_completed`, `l3_completed`, `failed`)
   - Batch summary accurately reflects mixed status
   - TD-PIPE-001 fix: Atomic batch completion check using `UPDATE...WHERE (SELECT count...)` pattern to prevent race condition

9. **AC9 — Rate limit handling (429):** When a provider returns 429:
   - The fallback runner tries the next provider in chain (rate limit may be provider-specific)
   - If all providers are rate-limited: re-throw to let Inngest retry with backoff
   - Inngest's exponential backoff: 1s, 2s, 4s (3 retries configured)
   - Other files in the batch continue independently (rate limit is per-provider, per step.run)

10. **AC10 — Audit trail for fallback events:** Every model switch is logged in `ai_usage_logs` with:
    - `model`: actual model used (fallback model)
    - New field tracking in audit: the `writeAuditLog` call for fallback events includes `{ action: 'ai_fallback_activated', metadata: { originalModel, fallbackModel, reason, layer } }`
    - Cost tracking continues for fallback models (different cost rates applied correctly via `estimateCost`)

11. **AC11 — Boundary values:**
    - Zero fallback models in chain → primary fails → throw directly (no loop)
    - Primary succeeds on first try → no fallback needed, `fallbackUsed: false`
    - Primary fails, first fallback succeeds → `fallbackUsed: true`, correct model in findings
    - All models fail → NonRetriableError after exhausting chain
    - File with 0 L1 findings + L2 fails → `ai_partial`, score = 100 (no penalties)
    - L2 partial failure (some chunks succeed, some fail) + L3 fails → L1 + partial L2 findings preserved
    - Retry on `ai_partial` file → only failed layers re-run, successful layer findings untouched
    - Retry on file that's already processing → reject with "File already being processed"
    - Batch with all files `ai_partial` → batch marked complete (terminal status)

## Dev Notes

### Architecture Patterns & Constraints

**Fallback chain consumption strategy:**
```
AI Call Attempt Flow:
  primary model → success → done
       ↓ fail
  classify error:
    rate_limit/timeout → try next model
    auth/schema/content → try next model
    unknown → re-throw to Inngest
       ↓ next model
  fallback[0] → success → done (mark fallbackUsed=true)
       ↓ fail
  fallback[1] → success → done (mark fallbackUsed=true)
       ↓ all exhausted
  rate_limit/timeout → re-throw (Inngest retries entire step)
  auth/schema/content → NonRetriableError (Inngest → onFailure)
```

**Partial results at handler level (Guardrail #13 compliant):**
```
// CORRECT — try-catch at handler level, NOT inside step.run
try {
  l2Result = await step.run('l2-...', () => runL2ForFile(...))
} catch {
  // L2 failed after all retries — set partial status
  await step.run('partial-l2-...', () => setPartialStatus(...))
}
```

**File status transitions (updated with ai_partial):**
```
uploaded → parsing → parsed → l1_processing → l1_completed
  → l2_processing → l2_completed → l3_processing → l3_completed
  → failed (pre-L1 failure)
  → ai_partial (L1 done but L2/L3 failed)
```

**ScoreBadge state transitions (complete picture with partial):**
```
L1 only       → 'rule-only'     (blue,   "Rule-based")
L1 + L2       → 'ai-screened'   (purple, "AI Screened")
L1 + L2 + L3  → 'deep-analyzed' (gold,   "Deep Analyzed")
Partial        → 'partial'       (orange, "Partial")          ← THIS STORY
Calculating    → 'analyzing'     (gray,   "Analyzing...")
```

**Fallback detection for badge rendering (client-safe):**
```typescript
// src/types/pipeline.ts — client-safe constants (NO server-only import)
export const PRIMARY_MODELS: Record<string, string> = {
  L2: 'gpt-4o-mini',
  L3: 'claude-sonnet-4-5-20250929',
} as const

// FindingListItem — check if finding used a fallback model
const isFallback = finding.aiModel !== null
  && finding.detectedByLayer !== 'L1'
  && finding.aiModel !== PRIMARY_MODELS[finding.detectedByLayer]
```

**Concurrency interaction — normal pipeline vs retry:**
- `processFilePipeline` and `retryFailedLayers` are separate Inngest functions with **independent** concurrency pools (both `limit: 1` per project)
- A retry could theoretically run simultaneously with a normal pipeline for the same project
- The CAS guard in helpers (`runL2ForFile` checks `status === 'l1_completed'`, etc.) prevents simultaneous mutation of the same file — duplicate invocations will fail CAS and throw `NonRetriableError`
- This is safe: at most one function can advance a file's status at a time

**Retry flow — which layers to re-run:**
```
File layer_completed='L1', mode='economy'   → retry L2
File layer_completed='L1', mode='thorough'  → retry L2 + L3
File layer_completed='L1L2', mode='thorough'→ retry L3 only
```

**Atomic batch completion (TD-PIPE-001 fix):**
```sql
-- Race-safe: only ONE concurrent processFile can mark batch complete
-- completed_at IS NULL ensures only first writer succeeds (sentinel pattern)
-- If RETURNING is empty → another invocation already completed OR not all files done
UPDATE upload_batches SET completed_at = NOW()
WHERE id = $1
  AND completed_at IS NULL
  AND (SELECT count(*) FROM files
       WHERE batch_id = $1
       AND status NOT IN ('l2_completed','l3_completed','ai_partial','failed')
  ) = 0
RETURNING *
```
