# Story 3.5: Score Lifecycle & Confidence Display

Status: review

## Story

As a QA Reviewer,
I want to see scores update progressively as each analysis layer completes, with per-finding confidence indicators calibrated to my language pair,
So that I can trust the scores, understand how they evolve, and make informed review decisions.

## Acceptance Criteria

1. **AC1 — Score calculating state:** When a file enters the pipeline, `scores.score_status` = "calculating" and the UI shows ScoreBadge with spinner + "Calculating..." text. The "Approve" button is disabled while score is calculating.

2. **AC2 — L1 score with AI pending indicator:** When Layer 1 completes and AI layers are pending, `scores.score_status` remains "calculating". ScoreBadge shows L1 score value with "Rule-based" layer badge (blue) AND an "AI pending" indicator alongside.

3. **AC3 — Layer completion score transitions:** When each subsequent layer (L2, L3) completes, the score transitions: dimmed state (`score_status='calculating'`) with "Recalculating..." badge visible, then Inngest recalculation (~1-2s), then new score pushed via Supabase Realtime with 200ms fade-in highlight. Dimmed state lasts until recalculation completes. The Approve button: `disabled = true` WHILE `score_status != 'calculated'`; `disabled = false` WHEN `score_status = 'calculated'` (FR22).

4. **AC4 — Finding change triggers recalculation:** When a reviewer changes a finding status (e.g., Reject), a `finding.changed` event is emitted after 500ms debounce. The Inngest recalculation runs in serial queue per project (`concurrency: { key: projectId, limit: 1 }`). During recalculation: previous score shown dimmed, "Recalculating..." badge, Approve button disabled. After completion: new score highlighted, button re-enabled.

5. **AC5 — Approve gate (server-side):** The Approve/Auto-pass Server Action verifies `scores.score_status` server-side before proceeding. Valid values allowing Approve: `'calculated'` or `'overridden'`. Values blocking Approve: `'calculating'`, `'partial'`, `'na'`, `'auto_passed'`. If status is "calculating", return `{ success: false, code: 'SCORE_STALE', message: 'Please wait for score recalculation to complete' }` per ActionResult<T> pattern (Architecture Decision 3.2).

6. **AC6 — Auto-pass rationale display:** When a file auto-passes after all layers complete, the rationale is displayed showing: final score with margin from threshold, finding counts by severity (Critical: 0, Major: X, Minor: Y), riskiest finding summary, and all criteria met checkmarks (FR23). Rationale is stored in `scores.auto_pass_rationale` as structured JSON string. **Note:** Epic AC says `jsonb` but actual DB column is `text` (per `src/db/schema/scores.ts`). No migration needed — store JSON as string, parse on read with safe fallback.

7. **AC7 — Confidence inline display:** In FindingListItem row, AI findings show confidence badge: "94%" with color coding: High (>=85%, green), Medium (70-84%, orange), Low (<70%, red). Findings below the applicable confidence threshold show "Below threshold" warning icon. **Note:** Inline ConfidenceBadge with color coding + L2 threshold warning already exists from Story 3.2c. Story 3.5 extends this to support L3 threshold (`l3ConfidenceMin`) — refactor `ConfidenceBadge` prop from `l2ConfidenceMin` to generic `confidenceMin` so the correct threshold is passed per layer.

8. **AC8 — Confidence expanded display:** In FindingListItem expanded detail area, the confidence badge includes a tooltip explaining language pair threshold: "EN->TH threshold: 93%" (sourced from `language_pair_configs`).

9. **AC9 — Confidence thresholds from language_pair_configs:** Confidence thresholds are sourced from `language_pair_configs.l2_confidence_min` for L2 findings and `language_pair_configs.l3_confidence_min` for L3 findings, matched to the file's language pair (FR36).

10. **AC10 — Threshold change handling:** When `language_pair_configs` threshold changes (admin adjusts), existing findings re-evaluate styling but NOT state. Toast: "Confidence thresholds updated". If finding confidence drops below new threshold, show "Below threshold" warning badge.

11. **AC11 — Fallback model confidence warning:** When a finding was generated by a fallback AI model, the confidence badge includes a warning icon and tooltip: "Generated by fallback model — confidence may need recalibration".

12. **AC12 — Boundary values:**
    - Score at exactly 95 (auto-pass boundary) with 0 critical -> auto-pass
    - Score at 94.99 -> review (not auto-pass)
    - Score at 70 (fail boundary) -> review
    - Score at 69.99 -> fail
    - Confidence at exactly 85% -> High (green)
    - Confidence at 84.99% -> Medium (orange)
    - Confidence at exactly 70% -> Medium (orange)
    - Confidence at 69.99% -> Low (red)
    - Auto-pass with 0 findings (score=100) -> show rationale with empty severity counts
    - Approve during calculating -> SCORE_STALE error
    - Approve when auto_passed -> ALREADY_APPROVED error (server-side defense guard — button hidden in UI, only reachable via direct API call or race condition)
    - L1 only with AI pending -> "AI pending" indicator visible
    - Threshold change from 80 to 90 -> findings between 80-89 gain "Below threshold" badge

### Key Design Decisions

**1. Approve action = gate mechanism only (Story 3.5 scope)**
The `approveFile.action.ts` validates the score gate and returns score details. It does NOT change file status or close review sessions — that's Epic 4 (Story 4-2: Core Review Actions). Story 3.5 establishes the server-side validation pattern that Epic 4 extends. Rationale: file approval requires review session infrastructure (tracking which findings were reviewed, who approved, approval notes) which is Epic 4 scope.

**2. Auto-pass rationale stored as JSON string in TEXT column**
`scores.auto_pass_rationale` is a `text` column (not jsonb). Story 3.5 stores structured JSON as a string and parses on read with safe fallback. No migration needed. If jsonb operators are needed later, that's a separate migration task. Existing plain-text rationale from scoreFile calls will gracefully degrade in the `AutoPassRationale` component (fallback to raw text display).

**3. Threshold change reactivity via Supabase Realtime**
Admin threshold changes are rare (not hot path). A lightweight Realtime subscription on `language_pair_configs` with 60s polling fallback is sufficient. No Zustand store for thresholds — React state in ReviewPageClient, passed as props to children.

**4. "AI pending" indicator is UI-only**
No new DB field needed. When `layerCompleted === 'L1'` and the file's processing mode expects AI layers (Economy or Thorough), the UI shows "AI analysis pending..." text. This is derived from existing state.

### Architecture Patterns & Constraints

**Score status lifecycle (complete picture):**
```
Pipeline starts     -> 'calculating' (spinner, Approve disabled)
L1 completes        -> 'calculating' (show L1 score, "AI pending")
L2 completes        -> 'calculating' -> brief recalc -> 'calculated'
L3 completes        -> 'calculating' -> brief recalc -> 'calculated' (final)
Auto-pass eligible  -> 'auto_passed' (rationale displayed, no Approve button)
Finding changed     -> 'calculating' (dimmed, "Recalculating...")
Recalc done         -> 'calculated' (highlight, Approve enabled)
AI failed           -> 'partial' (orange badge, retry button from Story 3.4)
Admin override      -> 'overridden' (Approve enabled)
```

**Approve button state matrix:**
| `score_status` | Approve button | Reason |
|---|---|---|
| `calculating` | DISABLED | Score not final |
| `calculated` | ENABLED | Ready for review |
| `partial` | DISABLED | AI incomplete |
| `overridden` | ENABLED | Admin confirmed |
| `auto_passed` | HIDDEN (show rationale) | Already approved by system |
| `na` | DISABLED | Cannot score |

**Score recalculation flow (existing infrastructure, Story 3.5 wires UI):**
```
Finding status change (UI)
  -> 500ms debounce (finding-changed-emitter)
  -> 'finding.changed' Inngest event
  -> recalculateScore function (serial queue per project)
  -> scoreFile() recalculates MQM
  -> Supabase Realtime push (scores table UPDATE)
  -> useScoreSubscription receives new score
  -> useReviewStore.updateScore() updates state
  -> ScoreBadge re-renders with animation
  -> Approve button re-enables
```

**Auto-pass structured rationale format (define in `src/features/scoring/types.ts`):**
```typescript
type AutoPassRationale = {
  score: number           // 96.5
  threshold: number       // 95
  margin: number          // 1.5
  severityCounts: {
    critical: number      // 0
    major: number         // 2
    minor: number         // 8
  }
  riskiestFinding: {
    category: string      // 'fluency'
    severity: string      // 'major'
    confidence: number    // 72
    description: string   // 'Awkward phrasing...'
  } | null                // null if 0 findings
  criteria: {
    scoreAboveThreshold: boolean  // true
    zeroCritical: boolean         // true
    allLayersComplete: boolean    // true
  }
  isNewPair: boolean      // false
  fileCount: number       // 51
}
```
