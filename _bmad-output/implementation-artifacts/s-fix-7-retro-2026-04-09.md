# Retrospective — S-FIX-7 Concurrent Review Soft Lock

**Date:** 2026-04-09
**Facilitator:** Bob (Scrum Master)
**Story:** S-FIX-7 (Phase 2 of UX/UI Debt Sprint)
**Outcome:** Done. Code verified end-to-end via Playwright MCP.

## Summary

S-FIX-7 took **5 code-review rounds + 1 E2E session** to ship, producing **~64 patches closing 9 CRIT and 20 HIGH bugs**, plus a spawned child story (S-FIX-7b, backlog). The initial implementation was functionally complete but had multiple defense-in-depth gaps and one critical workflow (the Release button) that was never actually tested in a real browser.

**The root cause of the bug volume was a single shortcut:** Task 9's "Multi-user verification via Playwright MCP" was marked done using SQL-based lock release instead of button clicks. That gap combined with 8 interlocking ACs created subtle interaction bugs that paper review could not reliably catch. Each CR round then introduced new bugs in the fixes, extending the loop.

## What worked well

### useGuardedAction helper extraction (R4.5)
Extracted a single hook owning the 6-layer mutation guard pattern (read-only → sync inflight ref → self-assign → action → try/catch → finally). Replaced duplicated pattern in 11+ call sites. The helper itself has 9 unit tests that caught the R5-M4 in-flight race assertion gap — dogfooding worked. Applied Mona's `feedback-rolling-bug-pattern` memory correctly, just later than ideal.

### Party-mode team consultation for M1 decision
When CR R2 surfaced a scope-expansion question (cron releasing `assigned` rows beyond AC4), a 4-voice team discussion (Mary Analyst, Winston Architect, John PM, Amelia Dev) gave three valid interpretations. Picked strict-spec compliance and spawned S-FIX-7b cleanly. Kept S-FIX-7 bounded.

### Playwright MCP E2E session at the end
Twenty minutes of real-browser verification caught a 404 URL bug (E2E-C1) that 5 paper-review rounds + 573 unit tests + type-check + lint all missed. Real clicks measured the "Approving..." pending state at 1700ms, proving R5-H1 fix was correct — unit tests couldn't have verified that.

### Acceptance Auditor inline fallback
When the third-agent layer failed with API overload (three separate rounds: R2, R4, R5), manually performing the spec-compliance audit inline provided ~70-75% coverage with documented question lists. Not ideal but prevented total coverage loss.

## What didn't work

### Task 9 "verification" was self-asserted without artifact
The initial feat commit explicitly deferred Task 9: *"Task 9 (Playwright MCP multi-user E2E verification) deferred — requires running dev server + Supabase instance."* The follow-up `1ec6779` marked Task 9 done but used SQL to release the lock instead of clicking the Release button. Three different CRIT bugs in the Release button flow (C4 self-assigned, R3-C1 admin-assigned, E2E-C1 404) originated from this single untested code path.

**This was the root sin.** Every downstream CR round was paper review fixing paper bugs because the authoritative real-browser verification never happened.

### Story complexity exceeded paper-review capacity
S-FIX-7 had 8 interlocking ACs where AC1 (self-assign) + AC3 (server defense) + AC6 (read-only UI) + AC7 (polling) + AC8 (race) all depended on each other. Guardrail #33 says split if AC coordinate >3 — we violated that. The complexity was above what paper review could reliably track across 5 rounds without introducing new bugs while fixing old ones.

### Pattern-matched fixes instead of tracing flow
R2-C1 privilege escalation: C4 widened `VALID_TRANSITIONS` to accept `cancelled`, but the server-side authorization gate wasn't updated. Two minutes of tracing "who can call this action with this new value" would have caught it. Instead the fix looked correct in isolation and introduced a new CRIT.

R5-H1 useTransition regression: mechanically replaced `startApproveTransition(async () => ...)` with `() => void guardedAction(...)`. Lint, type-check, and 573 unit tests all passed. Functional behavior broke because unit tests don't render the component and observe the pending state. Mechanical refactors need functional re-verification.

### Rolling-bug pattern applied too late
CR R1 found C1 (8 actions missing lock check). CR R2-R4 kept finding variations of "missing try/catch around self-assign" and "ref flipped after await". The helper extraction that would have killed the class (`useGuardedAction`) happened in R4.5 — **4 rounds after the pattern was first visible**. Applied Mona's memory correctly, just late.

### Acceptance Auditor API reliability
Failed in 3 of 5 rounds (R2, R4, R5) due to Anthropic API overload. When one of three review layers fails, coverage drops to 2/3 and spec-compliance bugs slip through. Inline fallback helps but isn't a substitute.

### 1 commit = many bug classes
The original feat commit bundled 9 ACs across 63 files. Smaller incremental commits with immediate verification would have constrained the blast radius of each bug.

## Root cause (5 Whys)

1. **Why did we need 5 CR rounds?** Each round found new bugs including some introduced by the previous round's fixes.
2. **Why did fixes introduce new bugs?** Paper review cannot trace runtime behavior — useTransition timing, URL resolution, React strict-mode double-mount.
3. **Why rely on paper review?** No E2E test infrastructure existed for the Release button flow. Task 9 was marked done without a real browser session.
4. **Why was Task 9 marked done without a real session?** The spec didn't require a test artifact — just a checkbox.
5. **Why didn't the spec require an artifact?** Story complexity (8 interlocking ACs) made a full E2E script feel like scope creep — easier to defer.

**Systemic root cause: Tasks in the spec allowed "verification" to be a self-asserted checkbox without a test artifact. That single loophole combined with high AC complexity created a gap paper review could not close.**

## Positive patterns to repeat

- Extract helpers for duplicated patterns (rolling-bug rule)
- Party-mode team consultation for scope-expansion decisions
- Spawn child stories (like S-FIX-7b) to capture gaps without blocking the parent
- E2E Playwright MCP session at the end of a story
- Dogfooding: helper with its own tests catches helper bugs
- Test data setup scripts (`sfix7-*.mjs`) that enable repeatable E2E runs
