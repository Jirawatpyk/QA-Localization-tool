## Challenges

### 1. Story 3.4 Scope Too Large

Story 3.4 (AI Resilience) had **11 ACs** — the most complex story in the project. Required 3 CR rounds and 119 tests. Combined fallback chain, partial results, retry logic, batch completion race condition (TD-PIPE-001), CAS guards — all in one story.

**Root Cause:** Resilience is inherently cross-cutting — touches AI providers, pipeline orchestrator, scoring, UI. Hard to split without creating forward dependencies.

**Action:** Cap at ≤7 ACs per story. If >7, must split before locking AC.

### 2. Sub-Story Proliferation (Wiring Gaps)

Epic 3 started with 6 planned stories (3.0-3.5) but shipped 14+ stories. Added mid-epic: 3.0.5, 3.1a, 3.1b, 3.2b5, 3.2b6, 3.2b7. All were "wiring gaps" — components existed but weren't mounted in pages.

**Root Cause:** Feature modules built components without page integration. Orphan scan only happened at epic boundary, not continuously.

**Action:** Run orphan scan every 5 stories, not just at epic sign-off.

### 3. Realtime Subscription Patterns Complex and Error-Prone

Three different Realtime patterns emerged, each with unique gotchas:

| Pattern | Hook | Gotcha |
|---------|------|--------|
| Score | `useScoreSubscription` | DELETE+INSERT ≠ UPDATE event (latent bug from 3.0, fixed in 3.2c) |
| Findings | `useFindingsSubscription` | Burst batching needed (N INSERTs → N events → queueMicrotask) |
| Thresholds | `useThresholdSubscription` | Filter by targetLang + debounce toast (500ms) |

**Root Cause:** No shared documentation of Realtime patterns. Each developer re-discovered gotchas independently.

**Action:** Create Realtime Subscription Pattern Guide before Epic 4.

### 4. CI-Specific Gotchas

- **dnd-kit mouse drag unreliable in headless browser** — CSS transforms shift rows during drag → bounding boxes stale. Solution: KeyboardSensor (`focus → Space → ArrowDown × N → Space`)
- **Prop sync anti-pattern (Guardrail #24)** — React 19 "adjust state during render" overwrites optimistic updates when RSC cache revalidates with stale data

**Root Cause:** Headless browser behavior differs from interactive; React 19 RSC cache behavior not fully documented.

**Action:** Both captured as guardrails (#24 + E2E gotchas doc). Will persist as institutional knowledge.

## Key Insights

1. **Proactive Guardrails from docs = zero preventable bugs** — writing guardrails BEFORE coding for new tech domains (AI SDK, Inngest L2/L3) eliminated the "first occurrence always slips through" pattern from Epic 2
2. **Prep Tasks are the highest-ROI investment** — P1-P5.1 created foundation that every story built on. Must continue for Epic 4 (keyboard spike, accessibility baseline)
3. **Scope ≤7 ACs per story** — Story 3.4 (11 ACs, 3 CR rounds) proves that large scope = more rounds, even with strong guardrails
4. **Realtime patterns need documentation** — 3 different patterns, 1 latent bug. Shared guide prevents re-discovery
5. **Orphan scan should be continuous, not one-time** — wiring gaps accumulate silently; catching them every 5 stories prevents sub-story proliferation
6. **Guardrail accumulation compounds** — 15 → 24 guardrails across 3 epics. Each is a real lesson encoded as a rule. This is the team's most valuable asset

## Epic 4 Gap Analysis Decisions (2026-03-08, Party Mode)

### Decision 1: Review Route Pattern — KEEP EXISTING

- **Existing:** `/projects/[projectId]/review/[fileId]/`
- **Epic 4 spec (incorrect):** `/projects/[projectId]/files/[fileId]/review/`
- **Decision:** KEEP existing route. Epic 4 story specs will be updated to match.
- **Rationale:** Route is fully wired (ReviewPageClient, 3 Realtime hooks, 2 server actions, E2E specs). Changing = pure churn. Existing pattern better for batch review extension.
- **Participants:** Winston (Architect), Amelia (Dev), Bob (SM)

### Decision 2: Severity-Major Contrast Token — CHANGE to amber-700

- **Current:** `--color-severity-major: #f59e0b` (amber-500, 2.9:1 contrast ❌)
- **New:** `--color-severity-major: #b45309` (amber-700, 5.4:1 contrast ✅)
- **Also:** `--color-status-pending` badges need dark text foreground (not white on amber)
- **Action:** Fix in Story 4.0 (`tokens.css` update + component audit)
- **Rationale:** WCAG 2.1 AA requires 4.5:1 for normal text (SC 1.4.3, Guardrail #26). amber-700 passes AA and still reads as "warning/amber" in severity hierarchy.
- **Participants:** Sally (UX Designer), Winston (Architect), Amelia (Dev), Bob (SM)
